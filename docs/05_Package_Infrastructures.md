## 1. Overview and Mental Model

Buildroot's **package infrastructure** is a set of Make macros that implement the
standard lifecycle for building and installing a third-party package:

```
download → extract → patch → configure → build → install (staging + target)
```

Each infrastructure implements this lifecycle for a specific build system.
You, the package author, only write a `.mk` file that fills in the *differences*
from the default behaviour — everything else is handled automatically.

```
ASCII: Buildroot Package Lifecycle

  ┌────────────┐     ┌──────────┐     ┌───────┐     ┌───────────┐
  │  download  │────▶│ extract  │────▶│ patch │────▶│ configure │
  └────────────┘     └──────────┘     └───────┘     └───────────┘
                                                            │
        ┌───────────────────────────────────────────────────┘
        ▼
  ┌───────┐     ┌──────────────────┐     ┌──────────────────┐
  │ build │────▶│ install-staging  │────▶│ install-target   │
  └───────┘     └──────────────────┘     └──────────────────┘

  Each step has:
    • A default implementation   (provided by the infrastructure)
    • Hook points                (*_PRE_*, *_POST_* variables)
    • Override functions         (PKG_BUILD_CMDS, PKG_INSTALL_TARGET_CMDS …)
```

## 3. autotools-package

### 3.1 When to use it

Use `autotools-package` when the upstream project uses the **GNU Build System**:
a `configure` script generated by Autoconf, and `Makefile.am` files processed by
Automake. Signs:

- Repository contains `configure.ac` or `configure.in`
- Tarball ships a pre-generated `configure` script
- Build instructions say `./configure && make && make install`

### 3.2 Minimal `.mk` file

```makefile
# package/libfoo/libfoo.mk

LIBFOO_VERSION  = 1.4.2
LIBFOO_SOURCE   = libfoo-$(LIBFOO_VERSION).tar.gz
LIBFOO_SITE     = https://example.com/releases
LIBFOO_LICENSE  = LGPL-2.1+
LIBFOO_INSTALL_STAGING = YES          # install headers/libs into staging/

LIBFOO_CONF_OPTS = \
    --disable-static \
    --enable-shared  \
    --without-python

$(eval $(autotools-package))
```

Buildroot will automatically pass the cross-compiler, sysroot, and host/target
triplets to `./configure`.

### 3.3 The C library being packaged

```c
/* src/foo.c  — the upstream C source being packaged */

#include "foo.h"
#include <stdlib.h>
#include <string.h>

struct foo_ctx {
    char   *name;
    int     value;
};

foo_ctx_t *foo_create(const char *name, int value)
{
    foo_ctx_t *ctx = malloc(sizeof(*ctx));
    if (!ctx) return NULL;

    ctx->name  = strdup(name);
    ctx->value = value;
    return ctx;
}

void foo_destroy(foo_ctx_t *ctx)
{
    if (!ctx) return;
    free(ctx->name);
    free(ctx);
}

int foo_get_value(const foo_ctx_t *ctx)
{
    return ctx ? ctx->value : -1;
}
```

```c
/* include/foo.h */

#ifndef FOO_H
#define FOO_H

typedef struct foo_ctx foo_ctx_t;

foo_ctx_t *foo_create(const char *name, int value);
void       foo_destroy(foo_ctx_t *ctx);
int        foo_get_value(const foo_ctx_t *ctx);

#endif /* FOO_H */
```

The upstream `configure.ac` fragment that tells Buildroot this is an autotools project:

```m4
# configure.ac  (upstream — not written by the package author)
AC_INIT([libfoo], [1.4.2])
AC_CONFIG_SRCDIR([src/foo.c])
AM_INIT_AUTOMAKE([-Wall -Werror foreign])
AC_PROG_CC
AC_CONFIG_HEADERS([config.h])
AC_CONFIG_FILES([Makefile src/Makefile include/Makefile])
AC_OUTPUT
```

### 3.4 Key hook variables

```makefile
# Pass extra flags to ./configure
LIBFOO_CONF_OPTS += --enable-threads

# Run commands after the default configure step
define LIBFOO_POST_CONFIGURE_FIXUP
    # Patch a generated header that hard-codes the build host path
    $(SED) 's|/usr/local|/usr|g' $(@D)/config.h
endef
LIBFOO_POST_CONFIGURE_HOOKS += LIBFOO_POST_CONFIGURE_FIXUP

# Extra environment variables forwarded to configure and make
LIBFOO_CONF_ENV += PKG_CONFIG_PATH="$(STAGING_DIR)/usr/lib/pkgconfig"
```

## 4. cmake-package

### 4.1 When to use it

Use `cmake-package` when upstream ships a `CMakeLists.txt`. CMake is the
de-facto standard for modern C and C++ projects. Signs:

- Top-level `CMakeLists.txt` present
- Build instructions say `cmake -B build && cmake --build build`
- Project uses `find_package()`, `target_link_libraries()`, etc.

### 4.2 Minimal `.mk` file

```makefile
# package/libbar/libbar.mk

LIBBAR_VERSION         = 2.1.0
LIBBAR_SOURCE          = libbar-$(LIBBAR_VERSION).tar.xz
LIBBAR_SITE            = https://github.com/example/libbar/releases/download/v$(LIBBAR_VERSION)
LIBBAR_LICENSE         = MIT
LIBBAR_INSTALL_STAGING = YES

LIBBAR_CONF_OPTS = \
    -DBUILD_SHARED_LIBS=ON   \
    -DBUILD_TESTING=OFF      \
    -DLIBBAR_ENABLE_DOCS=OFF

$(eval $(cmake-package))
```

Buildroot generates the CMake toolchain file automatically, setting
`CMAKE_C_COMPILER`, `CMAKE_SYSTEM_NAME=Linux`, and the sysroot.

### 4.3 The C++ library being packaged

```cpp
// src/bar.cpp  — upstream C++ source

#include <stdexcept>
#include <sstream>

Widget::Widget(std::string_view label, int id)
    : label_{label}, id_{id}

    std::ostringstream oss;
    oss << "Widget[" << id_ << "]: " << label_;
    return oss.str();
}

Registry& Registry::instance()
{
    static Registry reg;
    return reg;
}

void Registry::add(std::unique_ptr<Widget> w)
{
    widgets_.emplace(w->id(), std::move(w));
}

Widget* Registry::find(int id) const
{
    auto it = widgets_.find(id);
    return (it != widgets_.end()) ? it->second.get() : nullptr;
}

} // namespace bar
```

```cpp
#include <string_view>
#include <memory>

    Widget(std::string_view label, int id);

    [[nodiscard]] std::string describe() const;
    [[nodiscard]] int         id()      const { return id_; }
    [[nodiscard]] std::string_view label() const { return label_; }

private:
    std::string label_;
    int         id_;

class Registry {
public:
    static Registry& instance();

    void    add(std::unique_ptr<Widget> w);
```

## 5. meson-package

### 5.1 When to use it

Use `meson-package` when upstream uses **Meson** (with Ninja as backend). Meson
is popular in the GNOME/freedesktop ecosystem and is spreading to multimedia and
system libraries. Signs:

- `meson.build` at the project root
- `meson_options.txt` or `meson.options` present
- Build instructions say `meson setup build && ninja -C build`

### 5.2 Minimal `.mk` file

```makefile
# package/libbaz/libbaz.mk

LIBBAZ_VERSION         = 0.9.5
LIBBAZ_SOURCE          = libbaz-$(LIBBAZ_VERSION).tar.bz2
LIBBAZ_SITE            = https://gitlab.example.org/libbaz/-/archive/$(LIBBAZ_VERSION)
LIBBAZ_LICENSE         = Apache-2.0
LIBBAZ_DEPENDENCIES    = host-pkgconf zlib
LIBBAZ_INSTALL_STAGING = YES

LIBBAZ_CONF_OPTS = \
    -Dtests=false   \
    -Ddocs=disabled \
    -Dexamples=false

$(eval $(meson-package))
```

Buildroot generates a **cross file** (`meson --cross-file`) automatically.

### 5.3 The C library being packaged (Meson style)

```c
/* src/baz.c  — upstream C source using a Meson-driven build */

#include "baz.h"
#include <zlib.h>
#include <stdlib.h>
#include <stdio.h>

struct baz_compressor {
    z_stream zs;
    int      level;
};

baz_compressor_t *baz_compressor_new(int level)
{
    baz_compressor_t *c = calloc(1, sizeof(*c));
    if (!c) return NULL;

    c->level = level;
    if (deflateInit(&c->zs, level) != Z_OK) {
        free(c);
        return NULL;
    }
    return c;
}

int baz_compress(baz_compressor_t *c,
                 const uint8_t *in,  size_t in_len,
                       uint8_t *out, size_t out_cap,
                 size_t *out_len)
{
    c->zs.next_in   = (Bytef *)in;
    c->zs.avail_in  = (uInt)in_len;
    c->zs.next_out  = out;
    c->zs.avail_out = (uInt)out_cap;

    int rc = deflate(&c->zs, Z_FINISH);
    *out_len = out_cap - c->zs.avail_out;

    deflateReset(&c->zs);
    return (rc == Z_STREAM_END) ? 0 : -1;
}

void baz_compressor_free(baz_compressor_t *c)
{
    if (!c) return;
    deflateEnd(&c->zs);
    free(c);
}
```

The upstream `meson.build` that `meson-package` drives:

## 6. generic-package

### 6.1 When to use it

Use `generic-package` for everything that does **not** fit the three structured
build systems above:

- Hand-written or vendor `Makefile` with no standard configure step
- SCons, WAF, or other exotic build tools
- Shell-script-driven builds
- Pure header-only or data-only packages
- Pre-built binary packages

With `generic-package` you write **every lifecycle command** yourself using
override variables.

### 6.2 Minimal `.mk` file

```makefile
# package/myapp/myapp.mk

MYAPP_VERSION = 3.0.1
MYAPP_SOURCE  = myapp-$(MYAPP_VERSION).tar.gz
MYAPP_SITE    = https://example.com/dist
MYAPP_LICENSE = GPL-2.0-only

define MYAPP_BUILD_CMDS
    $(MAKE) $(TARGET_CONFIGURE_OPTS) -C $(@D) \
        CC="$(TARGET_CC)"     \
        CFLAGS="$(TARGET_CFLAGS)" \
        LDFLAGS="$(TARGET_LDFLAGS)"
endef

define MYAPP_INSTALL_TARGET_CMDS
    $(INSTALL) -D -m 0755 $(@D)/myapp \
        $(TARGET_DIR)/usr/bin/myapp
endef

$(eval $(generic-package))
```

The hand-written upstream `Makefile` (no `./configure`):

```makefile

CC      ?= gcc

LDFLAGS ?=
```

### 6.4 Advanced generic-package with all lifecycle overrides

```makefile
# package/myapp/myapp.mk  — full override example

MYAPP_VERSION = 3.0.1
```

## 7. Hook Variables and Override Functions

### 7.1 The naming convention

Every variable and function is prefixed with the **uppercase package name**:

```
LIBFOO_*    →  for package named  libfoo
MYAPP_*     →  for package named  myapp
```

### 7.2 Configuration option variables

| Variable | Used by | Purpose |
|---|---|---|
| `PKG_CONF_OPTS` | autotools, cmake, meson | Flags/options passed to configure |
| `PKG_CONF_ENV` | autotools, cmake, meson | Environment vars for configure |
| `PKG_MAKE_OPTS` | autotools | Extra `make` flags during build |
| `PKG_MAKE_ENV` | autotools | Environment vars for `make` |

```makefile
# autotools example
LIBFOO_CONF_OPTS += --with-ssl=$(STAGING_DIR)/usr
LIBFOO_CONF_ENV  += ac_cv_func_malloc_0_nonnull=yes   # cross-compile cache value

# cmake example
LIBBAR_CONF_OPTS += -DCMAKE_BUILD_TYPE=RelWithDebInfo
LIBBAR_CONF_OPTS += -DBAR_BACKEND=$(if $(BR2_PACKAGE_VULKAN_HEADERS),vulkan,opengl)

# meson example  (true/false, not ON/OFF)
LIBBAZ_CONF_OPTS += -Dlogging=true
LIBBAZ_CONF_OPTS += -Dbackend=$(if $(BR2_PACKAGE_GIO),gio,poll)
```

### 7.3 Hook variables (pre/post)

Hooks let you inject shell commands **before or after** any lifecycle step
without replacing the default implementation:

```
<PKG>_PRE_<STEP>_HOOKS   — run before the step
<PKG>_POST_<STEP>_HOOKS  — run after  the step
```

Available step names:
`DOWNLOAD`, `EXTRACT`, `RSYNC`, `PATCH`, `CONFIGURE`, `BUILD`,
`INSTALL_STAGING`, `INSTALL_TARGET`, `INSTALL_IMAGES`, `LEGAL_INFO`

```makefile
# Inject a pre-configure hook that patches a version string
define LIBFOO_FIX_VERSION_HEADER
    $(SED) 's/@VERSION@/$(LIBFOO_VERSION)/g' \
        $(@D)/include/foo_version.h.in > $(@D)/include/foo_version.h
endef
LIBFOO_PRE_CONFIGURE_HOOKS += LIBFOO_FIX_VERSION_HEADER

# Post-install hook: remove libtool .la files (never needed on target)
define LIBFOO_REMOVE_LA_FILES
    find $(STAGING_DIR)/usr/lib -name "*.la" -delete
endef
LIBFOO_POST_INSTALL_STAGING_HOOKS += LIBFOO_REMOVE_LA_FILES
```

### 7.4 Override functions (generic-package style)

Override functions **replace** the default implementation entirely.
They use the `define … endef` syntax and reference `$(@D)` (the build directory):

```makefile
# $(@D)           — package build directory (extracted source)
# $(TARGET_DIR)   — target root filesystem
# $(STAGING_DIR)  — staging area (headers + libs for cross-compilation)
# $(HOST_DIR)     — host tools directory
# $(MYAPP_PKGDIR) — directory containing myapp.mk (for bundled files)

define MYAPP_BUILD_CMDS
    $(TARGET_MAKE_ENV) $(MAKE) $(TARGET_CONFIGURE_OPTS) \
        -C $(@D) all
endef

define MYAPP_INSTALL_TARGET_CMDS
    $(INSTALL) -D -m 0755 $(@D)/myapp $(TARGET_DIR)/usr/bin/myapp
    $(INSTALL) -D -m 0644 $(MYAPP_PKGDIR)/myapp.service \
        $(TARGET_DIR)/usr/lib/systemd/system/myapp.service
endef
```

### 7.5 ASCII: hook and override anatomy

```
  Package lifecycle step  (e.g. CONFIGURE)
  ──────────────────────────────────────────────────────
  1.  Execute each function in  PKG_PRE_CONFIGURE_HOOKS
      │
      ▼
  2a. If PKG_CONFIGURE_CMDS is defined:
          execute PKG_CONFIGURE_CMDS           ← OVERRIDE (replaces default)
      else:
          execute default configure logic      ← DEFAULT  (autotools/cmake/meson)
      │
      ▼
  3.  Execute each function in  PKG_POST_CONFIGURE_HOOKS
  ──────────────────────────────────────────────────────

  Hooks   ADD behaviour  (default still runs unless overridden)
  Overrides REPLACE behaviour entirely
```

## 8. Cross-compilation Considerations

Buildroot's infrastructure auto-populates many variables for cross-compilation.
Understanding them helps write correct `CONF_OPTS` and override functions.

```makefile
# Key variables available in .mk files

$(TARGET_CC)         # Cross C compiler       e.g. arm-linux-gnueabihf-gcc
$(TARGET_CXX)        # Cross C++ compiler     e.g. arm-linux-gnueabihf-g++
$(TARGET_LD)         # Cross linker
$(TARGET_CFLAGS)     # CFLAGS for target      e.g. -march=armv7-a -mfpu=neon …
$(TARGET_CXXFLAGS)   # CXXFLAGS for target
$(TARGET_LDFLAGS)    # LDFLAGS for target
$(GNU_TARGET_NAME)   # Target triplet         e.g. arm-linux-gnueabihf
$(ARCH)              # Architecture           e.g. arm
$(STAGING_DIR)       # Staging sysroot        e.g. output/host/arm-linux-gnueabihf/sysroot

# Useful macro combining CC, CXX, LD, AR, etc.
$(TARGET_CONFIGURE_OPTS)

# Example: passing all of them to a hand-written Makefile
define MYAPP_BUILD_CMDS
    $(MAKE) $(TARGET_CONFIGURE_OPTS)         \
        CFLAGS="$(TARGET_CFLAGS)"            \
        CXXFLAGS="$(TARGET_CXXFLAGS)"        \
        LDFLAGS="$(TARGET_LDFLAGS)"          \
        PKG_CONFIG="$(PKG_CONFIG_HOST_BINARY)" \
        PKG_CONFIG_LIBDIR="$(STAGING_DIR)/usr/lib/pkgconfig" \
        -C $(@D)
endef
```

### 9.4 A more complete Rust library example

```rust
// src/lib.rs  — a Rust library crate (produces a .a for C FFI)

use std::ffi::{CStr, CString};
use std::os::raw::{c_char, c_int};

/// Process a string and return its length.
/// Exported for C FFI.
///
/// # Safety
/// `s` must be a valid null-terminated C string.
#[no_mangle]
pub unsafe extern "C" fn myrust_strlen(s: *const c_char) -> c_int {
    if s.is_null() {
        return -1;
    }
    CStr::from_ptr(s).to_bytes().len() as c_int
}

/// Reverse a C string, returning a newly allocated string.
/// Caller must free with `myrust_free_string`.
///
/// # Safety
/// `s` must be a valid null-terminated C string.
#[no_mangle]
pub unsafe extern "C" fn myrust_reverse(s: *const c_char) -> *mut c_char {
    if s.is_null() {
        return std::ptr::null_mut();
    }
    let input  = CStr::from_ptr(s).to_string_lossy();
    let rev: String = input.chars().rev().collect();

    match CString::new(rev) {
        Ok(cs) => cs.into_raw(),
        Err(_) => std::ptr::null_mut(),
    }
}

/// Free a string returned by `myrust_reverse`.
///
/// # Safety
/// `s` must have been returned by `myrust_reverse`.
#[no_mangle]
pub unsafe extern "C" fn myrust_free_string(s: *mut c_char) {
    if !s.is_null() {
        drop(CString::from_raw(s));
    }
}
```

```
  New package: which infrastructure to choose?
  ═══════════════════════════════════════════════════════════════════

  Does the upstream source have …

  ┌─ configure.ac  ──or──  configure  ──or──  Makefile.am ?
  │      YES  →  autotools-package
  │

  │      YES  →  cargo-package

  └─ None of the above  (hand Makefile, SCons, binary, data …) ?
```

## 11. Summary

### Quick-reference table

```
┌─────────────────────┬──────────────────────────────┬────────────────────────────────────────┐
│ Infrastructure      │ Key CONF_OPTS style           │ Mandatory overrides                    │
├─────────────────────┼──────────────────────────────┼────────────────────────────────────────┤
│ autotools-package   │ --enable-foo / --with-bar     │ None (configure auto-detected)         │
│ cmake-package       │ -DFOO=ON / -DBAR=value        │ None (CMakeLists.txt auto-detected)    │
│ meson-package       │ -Dfoo=true / -Dbar=enabled    │ None (meson.build auto-detected)       │
│ generic-package     │ N/A                           │ *_BUILD_CMDS + *_INSTALL_TARGET_CMDS   │
│ cargo-package       │ --features / --no-default-…   │ None (Cargo.toml auto-detected)        │
└─────────────────────┴──────────────────────────────┴────────────────────────────────────────┘
```

### Key takeaways

**1. Choose the infrastructure that matches the upstream build system.**
   Buildroot generates the correct cross-compile machinery (toolchain file,
   cross file, environment) automatically for autotools, cmake, and meson.

**2. Use `*_CONF_OPTS` for build-system option flags**, not override functions.
   Reserve override functions (`*_BUILD_CMDS`, `*_INSTALL_TARGET_CMDS`) for
   situations where the default command is simply wrong for the package.

**3. Hooks (`*_PRE_*_HOOKS`, `*_POST_*_HOOKS`) add behaviour without replacing it.**
   They are the cleanest way to apply patches, generate files, or clean up
   artefacts around any lifecycle step.

**4. `generic-package` gives full control but requires you to write everything.**
   Use `$(TARGET_CONFIGURE_OPTS)` and `$(TARGET_CFLAGS)` so the build respects
   the cross-compiler and optimisation flags Buildroot has configured.

**5. Rust packages use `cargo-package`**, which drives `cargo build` with
   `CARGO_BUILD_TARGET` set to the correct embedded target triple. Offline
   vendoring is handled automatically when `MYRUST_SITE_METHOD = cargo`.

**6. Always use `$(@D)` for the build directory** in override functions —
   never hard-code paths, because Buildroot manages out-of-tree builds.

```
  Buildroot output/
  ├── build/
  │   ├── libfoo-1.4.2/       ← $(@D) during libfoo build
  │   ├── libbar-2.1.0/

  ├── host/                   ← $(HOST_DIR)   cross-compiler, pkg-config …
```
